TIER 06

Operate — AI-Native Business Infrastructure

The previous tiers taught you to build. This one teaches you to operate. Workflows, orchestration, retrieval, evaluation, and deployment — the operator's stack for running a small team like a large organization.

📦 5 modules ⏱️ ~5 hrs 🎯 You leave with reusable workflows, not code samples
Modules
01

Outcomes, not features

The shift

Old internet era taught people how to code. The AI era rewards a different skill: knowing what to build. AI compresses implementation difficulty, which makes judgment, structure, and operational clarity the new bottleneck.

An operator does not start at syntax. They start at an outcome the business needs:

  • "I want inbound leads automatically organized."
  • "I want client reports generated each Monday."
  • "I want meeting notes turned into tasks."
  • "I want proposals drafted from a brief."

Each outcome is a system waiting to be designed. Code is downstream.

Naming the outcome well

A good operator outcome is specific, repeatable, and measurable. "Use AI more" is none of those. "Every inbound form submission produces a categorized record and a draft reply within 60 seconds" is all three.

Field test: if you can't sketch what "done" looks like for one full run of the workflow, the outcome isn't sharp enough yet.

Module artifact

Write three operator outcomes for your own work. Keep the ones that pass the field test.

02

Workflow design

What a workflow actually is

A workflow is a defined sequence of steps that turns an input into an output. Each step has: what it consumes, what it produces, who (or what) performs it, and what happens when it fails.

Workflows are the unit of leverage for small teams. Once a workflow exists on paper, the question shifts from "how do I do this?" to "which step do I automate first?"

The five questions for every step

  • Input: what arrives at this step?
  • Action: what transformation happens?
  • Output: what leaves this step?
  • Owner: human, model, or deterministic code?
  • Fallback: what happens when this step is uncertain or wrong?

Worked example: inbound lead workflow

Workflow sketch1. Form submission → raw message [deterministic] 2. AI summary → 2-line summary + intent [model] 3. Categorize → tag: hot / warm / spam [model + rules] 4. Draft reply → first-pass response [model] 5. Human review → approve / edit / reject [human] 6. Send + log → email out, row in CRM [deterministic]

Five of six steps are simple. Only step 5 needs a human, and only because trust isn't established yet. Once the model wins enough reviews in step 5, you tighten the rule and let some categories auto-send.

Module artifact

Pick one outcome from Module 1. Sketch the workflow as a numbered list. Mark each step deterministic / model / human.

03

Orchestration & tools

From prompt to system

Orchestration is what turns a pile of prompts into a system. It sequences steps, passes data between them, retries failures, and decides when to escalate to a human.

You don't need a new platform for this. You need a clear control flow — whether it's a spreadsheet trigger, a single script, or an existing automation tool you already pay for.

Tool calling — the bridge to action

Tool calling is the capability that lets a model invoke real functions: search a database, send an email, fetch a file. It is the difference between a model that talks about work and a model that does it.

Operator rule: wherever you'd rather have an exact answer than a confident guess, give the model a tool instead of asking it to estimate. Math, lookups, dates, IDs — all tools, never freehand.

Agents — when steps can't be scripted

An agent is a loop: plan, act, observe, repeat. Reach for one only when the steps genuinely can't be enumerated in advance (open-ended research, multi-tool investigations). For most operator workflows, a scripted sequence beats an agent on cost, latency, and reliability.

Module artifact

For your workflow from Module 2, list the tools each model step would need to call. If a step has no tool, ask whether it should.

04

Retrieval & grounding

The context window is finite

Everything a model knows about your task lives in its context window. Run out of room and earlier content drops out. Stuff it full of irrelevant material and the model loses focus. Context is real estate; spend it well.

Retrieval beats memorization

Retrieval is how you make AI work over your data — docs, CRM rows, policy text, past tickets — without retraining anything. The system pulls the relevant pieces in at the moment of the question and places them into the prompt.

Without retrieval, the model relies on training memory: confident, fast, occasionally wrong. With retrieval, you can cite the exact paragraph the answer came from.

Grounding and hallucination

Grounding ties the model's output to specific, verifiable sources. The opposite — confident output produced without a source — is a hallucination. Designing systems that ground by default and surface their sources is most of the work of building trustworthy AI infrastructure.

Cheap grounding wins: a single retrieved paragraph in the prompt often outperforms a more sophisticated setup. Start there.

Module artifact

For each model step in your workflow, list the documents or records it should retrieve from. If a step retrieves nothing, decide whether it should.

05

Evaluation & trust

You cannot ship what you cannot measure

Evaluation is how you find out whether a change to a prompt, model, or workflow made things better or worse. Without it, every tweak is vibes. With it, you have a record.

Three evaluation modes — pick what fits

  • Human review: a person rates outputs against a small fixed set of cases. Slow, accurate, irreplaceable for the first dozen runs.
  • Programmatic checks: deterministic rules — output is valid JSON, contains required fields, matches an expected length. Cheap, fast, catches obvious failures.
  • Model-graded: a second model rates the first one's output against a rubric. Scales further but needs spot-checking against humans.

Promotion through trust

New workflows start with humans reviewing every output. As evaluation evidence accumulates, you tighten the loop: model handles category A entirely, humans review category B, escalations only on category C. Trust is earned per category, not per workflow.

The school's claim, restated: the operator who can structure, evaluate, and tighten a workflow is more valuable than the engineer who can write any one of its steps.

Tier capstone

You have an outcome, a workflow, the tools each step needs, the documents each step retrieves from, and a plan for evaluating outputs. Run one workflow for one week with a human in every step. Then promote one category. Operating infrastructure exists when something works without you watching it.
